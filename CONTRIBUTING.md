# Contributing to DayZ Modding Archive

Thank you for your interest in contributing to the DayZ Modding Archive! This repository aims to be the most comprehensive resource for DayZ modding, server administration, and community tools.

## ğŸ“Œ Repository Policy: Main Branch Only

**This repository maintains all content in the `main` branch exclusively.** 

### Why Single-Branch?

- **Single Source of Truth**: All documentation, examples, and resources are in one place
- **Easy Navigation**: Users don't need to search across multiple branches for content
- **Simplified Maintenance**: Updates and contributions are straightforward
- **Complete Accessibility**: Every piece of information is immediately available from main

### What This Means for Contributors

- All pull requests should target the `main` branch
- All documentation, guides, and examples are maintained in `main`
- No feature branches contain content that isn't in `main`
- Content updates are merged directly to `main` after review

## ğŸ¤ How to Contribute

### Types of Contributions Welcome

We welcome contributions in the following areas:

1. **Documentation**
   - New guides and tutorials
   - Updates to existing documentation
   - Corrections and clarifications
   - Additional examples and use cases

2. **Code Examples**
   - Working implementations of DayZ features
   - Sample configurations (XML, JSON)
   - Script examples and templates
   - Killfeed implementations

3. **Resource Links**
   - Community tools and utilities
   - Helpful mods and plugins
   - Educational content and videos
   - Community servers and forums

4. **Corrections**
   - Bug fixes in documentation
   - Outdated information updates
   - Broken links
   - Typos and formatting issues

### Contribution Guidelines

1. **Fork the Repository**
   - Create a fork of this repository to your GitHub account
   - Clone your fork locally

2. **Create Your Changes**
   - Make your changes in your local fork
   - Ensure all changes maintain the existing documentation structure
   - Follow the formatting and style of existing documents
   - Test any code examples to ensure they work

3. **Submit a Pull Request**
   - Push your changes to your fork
   - Create a pull request targeting the `main` branch
   - Provide a clear description of your changes
   - Reference any related issues

4. **Code Review**
   - Maintainers will review your pull request
   - Address any feedback or requested changes
   - Once approved, your changes will be merged to `main`
